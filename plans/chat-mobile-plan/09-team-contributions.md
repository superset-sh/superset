## Phase 1 equivalent ‚Äî User Personas & Workflows

**Source:** conversation context + scope statement.

- **User persona:** A Superset team member who uses desktop chat as their primary chat surface today, runs work on remote/cloud hosts (not the local mobile device), and wants to monitor + respond to chat sessions away from a laptop.
- **Key pain points:** sessions stall mid-turn waiting for input mobile cannot provide (tool approval, ask_user, plan approval); cannot kick off remote work without a laptop; no way to monitor long-running agent turns on the go.
- **Key workflow:** user has an active session running on a remote host ‚Üí they leave their desk ‚Üí agent finishes a tool call and pauses ‚Üí user gets a push notification on phone ‚Üí user opens app, sees the pending approval, taps "Approve" from a bottom sheet ‚Üí session resumes.

## Phase 2 equivalent ‚Äî Architecture

**Source:** `plans/20260521-mobile-chat-research.md` (4 revisions on branch `local-setup-no-env`, ~3000 words). Key decisions:

- **Transport: mirror `apps/desktop/src/renderer/lib/host-service-client.ts` pattern.** Typed HTTP+tRPC client against `@superset/host-service` `AppRouter`, URL = relay-routed per active host. Confirmed by reading `host-service-client.ts:2,23` and the renderer CSP at `apps/desktop/src/renderer/index.html:14` (`%RELAY_URL%` whitelisted). Desktop already does this for v2 workspaces, project setup, agent configs, file reveal, etc.
- **Session listing: ElectricSQL.** `chat_sessions` is already exposed as an Electric shape filtered by org at `apps/electric-proxy/src/where.ts:136-137`. Mobile adds a collection in `apps/mobile/lib/collections/collections.ts`. Zero new infra.
- **Messages: in-memory on host runtime.** Confirmed via grep ‚Äî no `chat_messages` table, no `messages: jsonb` column anywhere in `packages/db/`, `packages/trpc/`, `packages/chat/`. Messages reach mobile via host-service tRPC (`chat.listMessages`, `chat.getSnapshot`, `chat.sendMessage`), not via DB.
- **Relay forwarding:** Hono + Upstash Redis + `jose` JWT (`apps/relay/src/{tunnel,directory,access,auth}.ts`). Per-host reverse WS tunnel registers in `tunnel.ts:56-121`; routing via Redis directory.
- **No tRPC subscriptions used today.** Confirmed via repo-wide grep on `.subscription(` ‚Äî chat is polling- and request-response-based on the relay path; streaming uses external DurableStreams via `apps/api /api/chat/[sessionId]/stream`. Decision deferred between SSE-through-relay vs DurableStreams for mobile streaming.

## Phase 3 equivalent ‚Äî UI Infrastructure

**Source:** `plans/20260521-mobile-chat-design-audit.md` (committed `f3e68314b`, 422 lines, generated by `frontend-designer` agent). Key findings:

- **Pattern: parallel implementations, not shared package.** Validated against `cadra-app/monorepo` (accessed via `gh` CLI; private repo) which has `packages/ui/ai-elements/*` (web shadcn) and `apps/mobile/src/components/ui/chat/*` (RN port) with shared component names + Tailwind + shadcn CLI conventions, zero code reuse at the JSX layer. Stack identical to Superset.
- **Compatibility:** ~80% of desktop chat's top-30 Tailwind classes work as-is under uniwind. All 20 semantic color tokens are present in `apps/mobile/global.css`. The ~20% remaining splits into mechanical translations (`space-y-* ‚Üí gap-*`, CSS `transition-* ‚Üí Reanimated`, `hover:* ‚Üí active:*`, `dark:* ‚Üí @variant dark`) and a small no-equivalent set (`group-*` selectors, `var(--radix-popover-trigger-width)`, `calc()`, `whitespace-pre-wrap`).
- **Port complexity (from audit ¬ß5):**
  - Light: `ChatInterface`, `UserMessage`, `AssistantMessage`
  - Medium: `MessageList` (FlashList swap), `MessagePartsRenderer` (markdown swap), `PendingApprovalMessage`, `PendingQuestionMessage`
  - **Heavy: `ChatInputFooter`** (Tiptap, drag-drop, file picker on desktop)
- **Designer's 2¬¢ baked into PRD:** bottom sheets > inline cards for pause prompts (UC-PAUSE-01/02/03); atomic streaming (NOT character-drip) for UC-RENDER-02; FlashList required for UC-RENDER-07; 44pt min hit targets in pause sheets; defer the warm-ember palette migration (keep mobile cool-neutral for v0).

## Open product decisions surfaced during research (not yet resolved)

These were flagged in the research artifact and remain open for `/kb-sprint-plan` to slot into specific sprints:

1. **JWT lifecycle** for mobile ‚Üí relay: per-call mint via cloud tRPC, device-held longer-lived host token, or server-side proxy that hides the relay from mobile. Affects sprint sequencing for transport setup.
2. **Live streaming transport**: SSE through relay (net-new), chunked HTTP through relay, or DurableStreams SSE (existing path). The "consider adding SSE to relay" idea from the team discussion lives here.
3. **Color palette alignment**: keep mobile cool-neutral for v0 (current default) or align to desktop's warm ember palette. Whole-app change, needs design sign-off ‚Äî flagged but not blocking v0.
4. **Mobile-side Tiptap WebView trade-off**: `@10play/tentap-editor` is the path taken (per user direction "if desktop is using tiptap we should use tiptap"). WebView overhead is real (~30-50MB memory, occasional keyboard quirks). Monitor in v0 and consider plain `TextInput` fallback if perf becomes a problem on mid-range Android.
