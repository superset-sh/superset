# Per-project Linear workspace linking

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

Reference: This plan follows conventions from AGENTS.md (root) and this template.


## Purpose / Big Picture

Today, a Superset organization can connect exactly one Linear workspace via OAuth, and that workspace implicitly applies to every project in the org. There is no way to point different projects at different Linear workspaces, and there is no way to connect more than one Linear workspace at all.

After this change, an admin can:

1. Connect **multiple Linear workspaces** to the same Superset organization (each via its own OAuth round-trip).
2. **Assign a single Linear workspace to a project** (one-to-one from the project's perspective).
3. **Reuse the same Linear workspace across multiple projects** (one workspace can back N projects).
4. Change or unassign the workspace for a project at any time.

A user can see this working by: (a) clicking "Connect another Linear workspace" on the integrations page and completing OAuth twice with two different Linear orgs; (b) opening a project's settings and picking which connected workspace it should sync tasks from; (c) selecting that project in the tasks view and using the inline workspace dropdown to either pick an existing workspace or kick off OAuth for a brand-new one ("Connect another workspace…") that auto-assigns to the selected project on callback; (d) creating a task in that project and seeing it appear in the chosen Linear workspace, not in the org's other connected workspace.

The two assignment surfaces — project settings and the tasks view — are equivalent: both call the same `setProjectConnection` mutation. The tasks-view dropdown is shown whenever a project is selected, regardless of connection count, so users always have a single discoverable place to wire up Linear from the surface where they're working with tasks.


## Assumptions

These are temporary planning assumptions. Each is either resolved in the Decision Log or removed by implementation end.

- We will keep the existing org-level `integrationConnections` table and the existing token-refresh code path. We are widening the table (relaxing a constraint) rather than introducing a parallel "workspace" table.
- The `v2Projects` table is the right place to store the project → workspace pointer (see Surprise #2 — the active desktop project model is `v2Projects`, not the v1 `projects` table). Tasks gain a `linearConnectionId` column in M6 to carry the workspace stamp directly.
- Existing organizations that already have one Linear connection should continue working without manual intervention. Existing projects without a workspace assignment should fall back to the org's single connection until an admin sets it explicitly.
- The Linear OAuth scope (`read,write,issues:create`) does not change.
- The Linear-to-Superset initial sync job (`/api/integrations/linear/jobs/initial-sync`) currently keys off `organizationId`. After this change it must run **per-connection**, not per-org, because an org can have multiple connections.
- Per the Decision Log: sync still fires immediately from the OAuth callback (preserving today's behavior), just scoped to the new connection rather than the whole org.
- `apps/desktop` consumes the same tRPC procedures the web app does. Desktop changes are limited to surfacing the new workspace picker; no new IPC channel is needed.

## Surprises & Discoveries

- Observation #1: The `tasks` table has no `projectId` column. Tasks today are scoped to `organizationId` only.
  Evidence: `packages/db/src/schema/schema.ts` — `tasks` table has `organizationId`, `assigneeId`, `creatorId`, `statusId`, but no `projectId`.
  Implication: For task-create-and-sync (`apps/api/src/app/api/integrations/linear/jobs/sync-task/route.ts`), we cannot route through `getLinearClient({ projectId })`. As a temporary measure, `getNewTasksTeamId` now picks the most-recently-updated connection deterministically; this means orgs with multiple Linear workspaces will see all sync-task creations land in whichever workspace was touched last. Real fix is a follow-up plan: add `projectId` to `tasks`, backfill, then route sync-task by project.

- Observation #2: There are two project tables — `projects` (v1 desktop) and `v2Projects` (v2 desktop, the active project model). The unified tasks/PRs/issues view added in commits `333874dd4`, `1210cca94`, `e73709baf` and the V2ProjectSettings page both use `v2Projects` exclusively.
  Evidence: `apps/desktop/src/renderer/routes/_authenticated/_dashboard/tasks/components/TasksView/components/TasksTopBar/components/ProjectFilter/ProjectFilter.tsx:28` — `q.from({ projects: collections.v2Projects })`. `apps/desktop/src/renderer/routes/_authenticated/settings/v2-project/$projectId/components/V2ProjectSettings/V2ProjectSettings.tsx:24` — same.
  Implication: The schema column belongs on `v2Projects`, not `projects`. The Decision Log entry for "where the column lives" is updated; the auto-generated migration must be regenerated.

- Observation #3: There is no project settings page in the web app. All project management lives in desktop (`apps/desktop/src/renderer/routes/_authenticated/settings/v2-project/...`). The web app only hosts integration connection management. The original plan's Milestone 5 wording referenced `apps/web/.../projects/...` settings — adjusted to target desktop.
  Evidence: `Glob apps/web/src/app/**/projects/**/page.tsx` returned no files.

- Observation #4 (user feedback): Integrations should be manageable directly from the desktop app, not just by deep-linking to web. Implemented as a new desktop sub-route `apps/desktop/src/renderer/routes/_authenticated/settings/integrations/linear/page.tsx`.


## Decision Log

- Decision: When a project has no explicit `linearConnectionId`, fall through to the org's Linear connection **only if exactly one is connected**. If the org has zero connections, Linear features are disabled for that project. If the org has 2+ connections, the project must have an explicit assignment; explicit assignments persist across future connect/disconnect changes.
  Rationale: Single-workspace orgs keep working with zero migration effort. Once an org has 2+ workspaces, the system stops guessing and the project↔workspace relationships set by the admin are durable: adding or removing other workspaces doesn't invalidate or rewrite existing assignments.
  Date/Author: 2026-05-08, AviPeltz

- Decision: Disconnecting a connection that is referenced by N projects is **not blocked**. Instead, surface a confirmation modal: "5 projects still use this workspace — would you like to reassign them first?" with two actions: "Reassign first" (closes the modal and surfaces a project list with quick reassignment controls) and "Disconnect anyway" (proceeds; those projects' `linearConnectionId` is set to null by the FK's `ON DELETE SET NULL`). No silent action; the user always opts in.
  Rationale: Blocking is too rigid for admins who want to clean up an unused workspace and accept that linked projects will need re-pointing. The warn-and-confirm pattern preserves user control while making the consequences clear.
  Date/Author: 2026-05-08, AviPeltz

- Decision: **No per-project team override in v1.** All projects pointing at the same connection share the connection's `newTasksTeamId`. If per-project team routing is needed, the workaround is to connect a distinct Linear workspace for that project (which is now supported).
  Rationale: A Linear workspace contains multiple teams (Frontend, Backend, Design, etc.), and a Linear issue must belong to exactly one team. We could let each Superset project pick its own team within a shared workspace, but it adds schema/UI complexity for a use case that distinct workspaces already cover. Defer until a user actually asks for it.
  Date/Author: 2026-05-08, AviPeltz

- Decision: A newly-connected workspace **initial-syncs immediately** from the OAuth callback. Subsequent assignments via `setProjectConnection` do not re-trigger sync.
  Rationale: User wants the workspace's issues available right away, including before any project is assigned (e.g., to browse Linear issues in the tasks view while deciding which project to assign). The cost of pulling issues for a workspace that's never assigned is acceptable.
  Date/Author: 2026-05-08, AviPeltz

## Context and Orientation

This work spans four pieces of the Superset monorepo:

- `packages/db` — Drizzle ORM schema definitions. The two relevant tables are `integrationConnections` (where Linear OAuth tokens live) at `packages/db/src/schema/schema.ts:174-218`, and `projects` at `packages/db/src/schema/schema.ts:366-396`. Type definitions for the JSONB `config` field live at `packages/db/src/schema/types.ts:1-10`.
- `apps/api` — The Next.js API app that hosts the OAuth routes. The connect route is `apps/api/src/app/api/integrations/linear/connect/route.ts` (1-54). The callback route is `apps/api/src/app/api/integrations/linear/callback/route.ts` (1-131). The callback exchanges an authorization code for tokens, fetches the Linear workspace (Linear calls these "organizations") via the SDK, and upserts a row in `integrationConnections`. There is a qstash-queued initial-sync job referenced at line 119 that today runs per-org.
- `packages/trpc` — Shared tRPC ("typed RPC") definitions. The Linear router at `packages/trpc/src/router/integration/linear/linear.ts` exposes `getConnection`, `disconnect`, `getTeams`, and `updateConfig`. The shared client factory `getLinearClient(organizationId)` at `packages/trpc/src/router/integration/linear/utils.ts:40-79` is the single point through which all Linear API calls flow.
- `apps/web` — The integrations UI lives under `apps/web/src/app/(dashboard-legacy)/integrations/linear/`. The "Connect Linear" button hits `apps/api/src/app/api/integrations/linear/connect?organizationId=…`. Project settings are under `apps/web/src/app/(dashboard-legacy)/projects/`.
- `apps/desktop` — Consumes the same tRPC client as web. The tasks view (recently unified per commits `333874dd4` and `1210cca94`) renders Linear tasks for a chosen project; once the resolver takes a project, the desktop app's existing "filter by project" UX automatically scopes to the right Linear workspace.

Terms used in this plan:

- **Linear workspace** — what Linear calls an "organization" in their API. Each Linear workspace has its own teams, issues, and labels. Distinct workspaces are isolated; an OAuth token is bound to the workspace the user picked at consent time. The fields `externalOrgId` and `externalOrgName` on `integrationConnections` capture the workspace's Linear ID and display name.
- **Superset project** — a row in the `v2Projects` table (the v2 desktop project model; the active surface in TasksTopBar's `ProjectFilter` and `V2ProjectSettings`). Currently bound to a GitHub repo (`repoCloneUrl`, `githubRepositoryId`). After this change, it will *optionally* also point at one Linear workspace via a new `linearConnectionId` foreign key.
- **Connection** — a row in `integrationConnections`. Today, the unique constraint `integration_connections_unique` on `(organizationId, provider)` forbids more than one Linear connection per org. We will replace this with a constraint that allows multiple Linear connections per org but forbids the same Linear workspace being connected twice to the same org.
- **tRPC procedure** — a named, typed RPC endpoint defined in `packages/trpc`, called from the web/desktop client via the generated tRPC client. Adding one is a matter of adding a `z.object(...)` input schema and a function to the router record.


## Plan of Work

The work breaks into six milestones. They are ordered for safe, incremental rollout: schema and resolver first (no user-visible change), then OAuth and tRPC (back end functional), then UI (user-visible), then sync-job cleanup. Each milestone leaves the system in a working state.


### Milestone 1: Schema changes

This milestone updates the database to allow multiple Linear connections per org and to add the project → connection pointer. No code paths consume the new column yet, so the system continues to behave exactly as before. After this milestone, `bun run typecheck` and `bun test` continue to pass; running the migration on a database with one existing Linear connection per org leaves that connection intact.

Edits:

1. In `packages/db/src/schema/schema.ts`, modify the `integrationConnections` table block (currently 174-218):
   - Remove the unique constraint `unique("integration_connections_unique").on(table.organizationId, table.provider)` at lines 207-210.
   - Replace it with `unique("integration_connections_org_provider_external_unique").on(table.organizationId, table.provider, table.externalOrgId)`. This permits N Linear connections per org but forbids reconnecting the *same* Linear workspace twice.
   - Keep the `integration_connections_org_idx` index.

2. In `packages/db/src/schema/schema.ts`, modify the `v2Projects` table block (currently around lines 404-434) — **NOT** the v1 `projects` table; the desktop tasks view and project settings both use `v2Projects` (see Surprise #2):

   - Add an index: `index("v2_projects_linear_connection_idx").on(table.linearConnectionId)` so the reverse lookup ("which projects use this connection?") is cheap; we'll need it for the disconnect-block check in Milestone 4.

4. The new constraint is `(organizationId, provider, externalOrgId)`. Existing rows have a non-null `externalOrgId` (set in the callback at line 96), so the new constraint is satisfiable for all existing rows. If any row has a null `externalOrgId` (e.g., from a partial OAuth in development), backfill it before applying the constraint or expect the migration to fail; document this in Surprises if encountered.

## Idempotence and Recovery

The schema migration is non-destructive: it adds a nullable column and replaces one unique constraint with a strictly weaker one. Re-running it is a no-op (drizzle-kit detects a clean state). If the migration fails partway:

- If the new unique constraint failed because some existing row has a null `externalOrgId`, backfill that row by re-running OAuth for that connection or by SQL-updating the row from Linear API data, then re-apply.
- The new column `projects.linear_connection_id` is nullable; rolling back is `ALTER TABLE projects DROP COLUMN linear_connection_id` followed by re-adding the original unique constraint. Do this only on a Neon branch, never on production.

The OAuth flow is naturally idempotent: re-connecting the same workspace updates the existing row (per the new conflict target). Re-running the initial-sync job is idempotent because tasks dedupe on `externalId`.

The new tRPC mutations are idempotent: setting a project's connection to the same value twice is a no-op; clearing an already-cleared project is a no-op.
