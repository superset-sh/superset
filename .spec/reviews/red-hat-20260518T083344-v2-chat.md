## Executive Summary

v2 makes the right architectural move: the Mastra runtime is lifted out of Electron main into a dedicated Bun HTTP service (`@superset/host-service`), and the renderer talks to it over HTTP via `workspaceTrpc`. Several specific v1 risks are genuinely fixed: the dual-query race (`getDisplayState` + `listMessages`) is gone — replaced by a single `getSnapshot`; the module-scope IPC singleton in v1's `ChatPane.tsx:18` is removed; renderer-controlled `cwd` is replaced by a DB-resolved `worktreePath`; and v1's 60 fps polling override is back down to 4 fps.

**But the migration created five new, serious problems and inherited or worsened four others.** The most urgent:

1. **`abort: async () => undefined` (useWorkspaceChatDisplay.ts:289)** — a Category-3 semantic stub. The command is exposed as a public API and silently does nothing. Both renderer and mastra reviewers flagged this independently.
2. **`getMcpOverview` returns a hardcoded `{ sourcePath: null, servers: [] }`** stub at `packages/host-service/src/runtime/chat/chat.ts:793-798`. The `never[]` return type is the implementer flagging their own stub.
3. **Process-global `process.env` credential mutation in a multi-tenant Bun process** (`packages/host-service/src/providers/model-providers/utils/runtime-env/runtime-env.ts:12-19`) — **strictly worse than v1**. v1's TOCTOU race was bounded by one active chat at a time per Electron main; v2 runs N concurrent sessions across M workspaces on a single Bun process with no mutex. Session A's credentials can authenticate Session B's turn.
4. **`hostServiceCoordinator.restart` and `reset` are `publicProcedure`** with no caller-org validation (`apps/desktop/src/lib/trpc/routers/host-service-coordinator/index.ts:37-59`). Any compromised renderer can SIGKILL any organization's host-service.
5. **`notifications.hook` is intentionally `publicProcedure`** (`packages/host-service/src/trpc/router/notifications/notifications.ts:52-54`) with `terminalId: z.string().optional()` (no UUID validation). Combined with the relay path, this is enumeration + cross-workspace event injection.

And new in v2: pane unmount orphans the host-service runtime (no `stop()` on cleanup), `disposeRuntime` doesn't run SessionEnd hooks, session title generation has been silently dropped (feature regression vs v1), `no-electron-coupling.test.ts` passes but never walks the `@superset/chat/server/desktop` import graph (false-green architectural invariant), and there is a session-bootstrap split-brain: cloud creates the session record before host-service is told about it.

The v1 stack (`chatRuntimeService`, `chatService`) is **still registered and still active** at `apps/desktop/src/lib/trpc/routers/index.ts:35-36`. v1 is not deprecated; v2 is opt-in. The two stacks share the same preload, so v1's CRITICAL-1 (open `ipcRenderer` allowlist) applies unchanged to v2 panes.

## HIGH Confidence Findings (multi-reviewer consensus or CRITICAL from domain specialist)

### V2-H1. `abort` command is a Category-3 stub *(BOTH renderer + mastra reviewers — independent agreement)*
- **Location**: `apps/desktop/src/renderer/.../useWorkspaceChatDisplay/useWorkspaceChatDisplay.ts:289`
- **Evidence**: `abort: async () => undefined` — literal no-op exposed on a public commands API. The v2 host-service router has no `abort` procedure; there is only `stop`. Renderer reviewer confirms `useSlashCommandExecutor` (line 85) does call `commands.stop()` not `commands.abort()`, so the executor is clean — but `abort` remains on `UseChatDisplayReturn`'s public contract. Any future caller, slash command, or test reaching for it silently no-ops.
- **Why this is the SUPREME RULE violation**: this is the canonical "stubbed core logic" pattern — a function signature that promises behavior with an empty body. Per CLAUDE.md: *"Any agent that stubs required behavior … will lose the ability to operate."*
- **Fix**: Either delete `abort` from the returned commands object (TypeScript will surface callers) or alias it to `stopMutation.mutateAsync(queryInput)`. Both are 5-line changes. Stop shipping a function that lies about its capability.

### V2-H2. `getMcpOverview` returns a hardcoded stub
- **Location**: `packages/host-service/src/runtime/chat/chat.ts:793-798`
- **Evidence**:
  ```ts
  async getMcpOverview(_input): Promise<{ sourcePath: string | null; servers: never[] }> {
    return { sourcePath: null, servers: [] };
  }
  ```
  Exposed as `protectedProcedure` (router `chat.ts:165-169`). The `never[]` return type is the implementer flagging it. MCP UI gates on `servers.length` and will silently show "no servers" even if the workspace has MCP servers configured.
- **Fix**: Either remove the endpoint, return an explicit `{ disabled: true }` discriminated union the renderer respects, or implement the introspection. Don't ship a `never[]` in production.

### V2-H3. Process-global `process.env` credential mutation — *worse than v1* in a multi-tenant Bun process
- **Location**: `packages/host-service/src/providers/model-providers/utils/runtime-env/runtime-env.ts:12-19`, called from `LocalModelProvider.prepareRuntimeEnv()` (line 60-66), invoked per-session at `packages/host-service/src/runtime/chat/chat.ts:464`
- **Evidence**: `delete process.env[key]` then `process.env[key] = value`, with no mutex. `LocalModelProvider` is a singleton on the host-service. v1 had this issue inside Electron main (single-active-chat-by-UI-design). v2 has N concurrent sessions across M workspaces with no synchronization. Concurrent `prepareRuntimeEnv` calls race: Session A's Anthropic key can authenticate Session B's mid-flight turn; subprocesses spawned during the delete-then-set window inherit a torn env.
- **Impact**: Multi-tenant credential blast radius. In the worst case, an enterprise user with two orgs configured runs Workspace-1's Mastra agent against Workspace-2's API key. From the user's perspective: API errors, confusing "not authorized" messages, and most insidiously, billing leaks across credentials.
- **Fix**: Scope credentials to `createMastraCode` per-invocation options instead of mutating global env. If `mastracode` doesn't support that, wrap each model call in a subprocess with its own env namespace, or at minimum serialize all `prepareRuntimeEnv` calls through a per-process mutex.

### V2-H4. `hostServiceCoordinator.restart` and `reset` are `publicProcedure` — any renderer can SIGKILL any org's host-service
- **Location**: `apps/desktop/src/lib/trpc/routers/host-service-coordinator/index.ts:37-59`
- **Evidence**: Both procedures use `publicProcedure` with `organizationId: z.string()`. No check that caller's authenticated session matches `input.organizationId`. `reset` escalates to `killProcess(manifestPid, "SIGKILL")`. A compromised renderer (XSS, supply-chain compromise, dev-mode injection) can loop calls across every visible `organizationId` to deny service to all workspaces.
- **Fix**: Add a coordinator-level middleware that asserts `session.activeOrganizationId === input.organizationId`. Or split into an `adminProcedure` and gate destructive ops behind it.

### V2-H5. `notifications.hook` intentionally unauthenticated; `terminalId` is unvalidated
- **Location**: `packages/host-service/src/trpc/router/notifications/notifications.ts:18-22,52-54`
- **Evidence**: `publicProcedure` with comment "Intentionally unauthenticated: a caller can only trigger a chime and a sidebar indicator." Input `terminalId: z.string().optional()` (no UUID format check). Response differs between found/not-found via `ignored: true|false` field — terminal-existence oracle. Reachable through the relay (the tunnel forwards regardless of procedure auth level).
- **Impact**: Cross-workspace event injection, terminal-ID enumeration, UI confusion (forged agent lifecycle events from external sources). The "only a chime and sidebar indicator" framing understates: agentic lifecycle UI signals user trust state.
- **Fix**: Require a short-lived HMAC token scoped to the workspace's terminal, OR move the hook auth check to validate against the relay-injected user identity. At minimum: `z.string().uuid()` validation.

### V2-H6. Preload `ipcRenderer` open-channel relay — inherited from v1, still applies to v2 windows
- **Location**: `apps/desktop/src/preload/index.ts:34-59`
- **Evidence**: `window.ipcRenderer.invoke/send/on/off` forwards any channel string. The same preload runs for v2 BrowserWindows. v2 reduces dependence on Electron IPC for chat (HTTP transport instead), but the v1 surface (`chatRuntimeService`, `chatService`) is still registered in the AppRouter and reachable through the open relay from any v2 renderer code.
- **Fix**: `ALLOWED_CHANNELS = new Set([...])` allowlist in preload. Single-file fix that benefits both v1 and v2.

### V2-H7. Pane unmount during active turn orphans the host-service runtime *(NEW in v2)*
- **Location**: `apps/desktop/src/renderer/.../ChatPaneInterface.tsx` (no cleanup effect for active turn) + `packages/workspace-client/src/workspace-trpc.ts:5` (`abortOnUnmount: true`)
- **Evidence**: `abortOnUnmount: true` on `workspaceTrpc` cancels the in-flight HTTP `sendMessage` mutation when the pane unmounts. The host-service receives a dropped TCP connection — it does not interpret this as a cancellation signal for the Mastra agent. The agent keeps running. No `commands.stop()` is called from `ChatPaneInterface`'s unmount path (the cleanup at line 739-744 only clears retry timers). v1 didn't have this exact bug because the in-process IPC tied lifecycle to the runtime; v2's HTTP boundary breaks that.
- **Impact**: Closing a pane during agent work leaves Mastra running indefinitely, consuming CPU and potentially executing tool calls (file writes, bash) after the user has dismissed the UI. In dev mode this means observing "the agent kept editing files after I closed the tab."
- **Fix**: Add a `useEffect` cleanup in `ChatPaneInterface` (or the controller) that fires `commands.stop()` when `isRunning === true` at unmount.

### V2-H8. `disposeRuntime` does not fire SessionEnd hook or call `harness.destroy()` *(NEW in v2)*
- **Location**: `packages/host-service/src/runtime/chat/chat.ts:537-572`
- **Evidence**: Calls `runtime.harness.abort()` and `mcpManager.disconnect()`. Does not invoke `hookManager` SessionEnd. Does not call `harness.destroy()`. The harness's event subscription registered at chat.ts:385 holds a closure over `runtime` — never released even after `this.runtimes.delete(sessionId)`. Per-workspace SessionEnd hooks configured in `.claude/settings.json` never fire.
- **Fix**: In `disposeRuntime`, before delete: `await runtime.hookManager?.runSessionEnd().catch(() => {})` and `await runtime.harness.destroy?.().catch(() => {})`. Add an idle-TTL sweep that calls `disposeRuntime` for sessions untouched in N minutes.

### V2-H9. Session title generation is missing — feature regression vs v1 *(NEW in v2)*
- **Location**: `packages/host-service/src/runtime/chat/` (no `generateTitle` or `updateTitle` calls anywhere); v1 reference: `packages/chat/src/server/trpc/utils/runtime/runtime.ts:466-537`
- **Evidence**: v1's `sendMessage` path fired `void generateAndSetTitle(...)` after every send, called `api.chat.updateTitle.mutate(...)` on the 1st and every 10th user message. v2's `sendMessage` (chat.ts:676-698) returns immediately. The host-service tRPC chat router has no `updateTitle` call. The cloud `chat.updateSession.mutate({ lastActiveAt })` (router lines 69-71) updates only activity, not title.
- **Impact**: v2 sessions never auto-title. Session selector shows "New Chat" forever. Possibly intentional if title generation is being moved cloud-side, but no evidence of that — and silent regressions like this corrode user trust.
- **Fix**: Decide ownership (host-service vs cloud API) and implement. If owned by host-service, port `generateAndSetTitle` and wire it to `ctx.api.chat.updateTitle`. If cloud-owned, document the deferment.

### V2-H10. Cloud/host-service session bootstrap race
- **Location**: `apps/desktop/src/renderer/.../useWorkspaceChatController.ts:41-49,129-157`, then `apps/desktop/src/renderer/.../ChatPaneInterface.tsx:529-545,692`
- **Evidence**: `createSessionRecord` calls cloud API (`apiTrpcClient.chat.createSession.mutate`) then immediately returns. The caller fires `sendMessage` against host-service. There's no two-phase commit — if host-service is offline, the cloud session is orphaned. If `sendMessage` fails for any reason, the cloud record persists. No reconciliation path.
- **Fix**: Make session creation lazy on the host-service side (so the first `sendMessage` bootstraps), or require a `host-service.ensureSession` step before sending and only persist to cloud after host-service ack.

### V2-H11. Cross-session ownership gap — any window can act on any session
- **Location**: `packages/host-service/src/runtime/chat/chat.ts:493-521`; router `chat.ts:57-79`
- **Evidence**: `getOrCreateRuntime(sessionId, workspaceId)` matches sessionId to workspaceId but doesn't verify the *caller* owns the workspace. PSK auth (`protectedProcedure`) only checks the shared host-service secret — same for all windows on the same org. Window A (in workspace W1) can issue `chat.sendMessage({ sessionId: S2, workspaceId: W2 })` and act on Window B's session.
- **Impact**: Single-user desktop is low-exploit, but a compromised renderer in any v2 window has full access to all sessions across all open workspaces.
- **Fix**: Encode caller identity in `protectedProcedure` middleware; reject when the resolved workspace doesn't match the renderer's claim (or scope PSK per-workspace).

### V2-H12. 4 fps polling with `refetchIntervalInBackground: true` and no idle throttle
- **Location**: `useWorkspaceChatDisplay.ts:115,120-126`
- **Evidence**: Default `fps: 4` (250ms interval). `refetchIntervalInBackground: true`. No adaptive throttle when `!isRunning`. With 5 open panes: 20 HTTP req/sec to host-service indefinitely, including when window is minimized.
- **Impact**: Battery drain, accelerated host-service request queue, masks `staleTime: 5000ms` set in `WorkspaceClientProvider.tsx:7` (data is always fresher than stale at 4fps).
- **Fix**: Adaptive: `fps: isRunning ? 4 : 1` (or 0.5). Drop `refetchIntervalInBackground` unless explicitly required.

### V2-H13. Synchronous Drizzle workspace lookup on the host-service event-loop
- **Location**: `packages/host-service/src/runtime/chat/chat.ts:453-455,756-758`
- **Evidence**: `resolveWorkspaceCwd` uses `.sync()` on a Drizzle SQLite query. Called on every `getSlashCommands` / `resolveSlashCommand` / `previewSlashCommand` — which are themselves called from the slash-command UI on every input edit.
- **Impact**: Synchronous SQLite blocks the Bun event loop for every other in-flight tRPC request for the duration of the disk I/O. Multiplies with H12's polling.
- **Fix**: Replace `.sync()` with async; cache resolved cwd on `RuntimeSession`.

### V2-H14. `no-electron-coupling.test.ts` does not walk the import graph — false-green architectural invariant
- **Location**: `packages/host-service/src/no-electron-coupling.test.ts:18`, vs `packages/host-service/src/runtime/chat/chat.ts:6-8`, `packages/host-service/src/app.ts:4`
- **Evidence**: Test walks only `.ts` files inside `packages/host-service/src/` for direct `from 'electron'` / `BrowserWindow` / `ipcMain` patterns. Both `chat.ts` and `app.ts` import from `@superset/chat/server/desktop`, which barrel-exports `ChatService` (which in turn calls `createAuthStorage` from `mastracode` and orchestrates OAuth loopback servers — code designed for a process that owns UI). The test passes while the architectural intent is violated transitively.
- **Fix**: Add `@superset/chat/server/desktop` to `ELECTRON_PATTERNS`, or replace with a build-artifact check that verifies the bundled host-service output has zero Electron references. The slash-command helpers being imported are pure functions — they should be extracted into a separate package that doesn't ship `ChatService`.

### V2-H15. `app.ts` instantiates `ChatService` inside host-service — architecturally suspect
- **Location**: `packages/host-service/src/app.ts:4,117`; `packages/host-service/src/types.ts:16`
- **Evidence**: `const chatService = options.chatService ?? new ChatService();` `ChatService` manages OAuth loopback HTTP servers and `mastracode`'s auth storage. Designed for a process that owns a UI and can redirect a browser. Inside host-service (Bun, possibly headless / remote-tunnel-only), the loopback flows can't actually complete. AND: two paths now write to `mastracode`'s shared auth storage (ChatService via host routes + `LocalModelProvider` via `resolveAnthropicCredential`). Last writer wins.
- **Fix**: Move auth/OAuth orchestration to the desktop main process; have host-service consume *resolved* credentials over the IPC handshake. ChatService doesn't belong in host-service at all.

## Contradictions and Debate

- **Workspace cwd path traversal**: electron-reviewer rated this **FIXED in v2** (cwd is DB-derived from `workspaces.worktreePath`). Mastra-reviewer didn't re-flag it. **Agreement: genuinely fixed.**
