Customer Tracking Portal
architecture & build notes

How the Superset team sees every customer — company by company, domain by domain, person by person — with paying status, live product activity, health scoring, and AI-researched context.

01

Overview

The portal is a Vite single-page app (apps/customers, port 3005) gated to @superset.sh team members. It answers five questions about any customer: who are they, do they pay, how active are they, are they slipping away, and what should we know before reaching out.

Five surfaces, each one drill-down deeper than the last:

SurfaceRouteWhat it shows
Companies/companiesOrgs with plan, seats, 30d events + trend, last active, health & stage — filterable, searchable
Company detail/companies/$orgIdSubscription card, weekly activity chart, members with per-user stats
Domains/domainsUsers rolled up by email domain (finds multi-user companies without a formal org) + pinned tab
Domain detail/domains/$domainAggregate stats, AI company card, activity chart, per-user table with researched roles
User detail/users/$userIdProfile, orgs, activity, surface split, AI-researched role + social links

Design constraint honored throughout v1: no new database tables, no cron jobs. Everything is computed live from existing data plus layered caches, so the feature shipped without a migration.

The SPA holds no secrets and does no data access of its own — every byte flows through the customers tRPC router in packages/trpc, where adminProcedure enforces the team-only rule server-side. The frontend's session check is UX, not security.

The app deliberately reuses the monorepo's existing rails: @superset/ui for components, @superset/auth for sessions, the shared PostHog query client with its hourly cache, and the same NEXT_PUBLIC_* env vars every other app reads (exposed to Vite via envPrefix).

03

Data pipeline

The core join: companies and money live in Postgres; behavior lives in PostHog. They meet on distinct_id = users.id — every identified PostHog person carries our user UUID.

The activity snapshot

One HogQL query computes per-user aggregates over the last 90 days of 12 curated core events (workspace_opened, chat_message_sent, cli_command_invoked, …). It deliberately excludes $pageview — anonymous posthog-js IDs are also UUIDs and would flood the join — and funnel events like onboarding_*, which signal intent, not engagement.

  • Per user: last event timestamp, 7d / 30d / prior-30d counts, active days, and per-surface counters (desktop / CLI / chat).
  • ORDER BY last_ts DESC LIMIT 65000 — PostHog clamps at ~50k rows, so any truncation drops only the longest-dormant users.
  • Validated against production: a known user's dashboard last-active matches PostHog ground truth to the millisecond; 96.7% of core events attribute to real user ids.

The domain index is the workhorse for the Domains surface: a memoized walk of ~50k users grouping activity, org membership, and counts per email domain (freemail included; filtered at query time). Requests only filter / sort / paginate it — which is why a domain search resolves in ~30ms.

04

Health & stage model

Both models are pure, tested functions in packages/shared — no state, no I/O, thresholds are single constants.

Health — recency of core activity

TierLast core eventMeaning
Active≤ 7 daysUsing the product this week
Idle8–14 daysMissed a week — light nudge territory
Cooling15–30 daysGenuinely drifting
Dormant> 30 days / neverLapsed; paying + dormant = churn risk

Stage — adoption footprint

Solo (1 user) → Team (2–9) → Scale (10–49) → Enterprise (50+ users at the domain, or an enterprise plan). This measures footprint inside Superset; true firmographics come from enrichment and can differ.

Trend everywhere means one thing: core-event volume, last 30 days vs the 30 days before — spelled out in every tooltip, with a "no baseline" state when the prior window was empty.

05

AI enrichment

Firmographics and people-context are researched on demand, never scraped ahead of time. Two backends, split by what a 16-lookup head-to-head benchmark showed each is best at:

People → Exa/search + outputSchema, category: people · ~2s · ~1–2¢ · near-100% hit rate · grounding citations for free
│
Companies → Claudeopus + server-side web_search · 30s–2min · catches acquisitions & late rounds, avoids wrong-entity hits
──▶both land in
KV cache30-day TTL · empty results retry after 1 day (search variance)
  • Manual by default. Nothing calls an AI until a human clicks "Research" — but cached results always render instantly via cache-only read procedures.
  • Per-domain auto mode. A switch fires a one-shot background batch (company + 200 most-active users, concurrency 8, ~1 minute) with a persisted progress bar. New signups at auto domains get caught up on the next page view. Re-research always requires a manual action.
  • Anti-hallucination: prompts demand null over guesses; person research must match name and employer domain; results carry confidence + source links, labeled "AI-researched".
  • Each backend falls back to the other if its API key is missing.
Benchmark note. Exa is 10–200× faster and ~5× cheaper but occasionally returns a confident wrong entity (it once answered a Brazilian research lab for railway.app). Claude is slow but current (knew Loom was acquired by Atlassian). Hence the split — and why company research tolerates the wait.
06

Auth & access

  • Server truth: every procedure sits behind adminProcedure, which rejects any session whose email doesn't end in @superset.sh.
  • Client gate: the SPA checks the better-auth session on load and bounces signed-out visitors to the web app. Cookies are scoped to .superset.sh, so one sign-in covers all subdomains.
  • CORS: being a browser SPA (unlike the server-rendered admin app), the API had to allowlist the customers origin — NEXT_PUBLIC_CUSTOMERS_URL plus the dev port, in both the CORS proxy and better-auth's trusted origins.

The staleness trade is explicit: lists can lag reality by up to 15 minutes (activity by up to an hour). The Refresh button busts all server memos on demand, and every table shows a dim + spinner pending state during refetches so sub-second waits never look frozen.

09

Deployment (planned)

Vercel via CLI in the superset-sh team: build the SPA locally in the monorepo with production env baked in (Vite inlines at build time), deploy the static output, assign customers.superset.sh. The API project then needs NEXT_PUBLIC_CUSTOMERS_URL (activates CORS), a production POSTHOG_PROJECT_ID, and the permanent ANTHROPIC_API_KEY + EXA_API_KEY.

Known follow-ups. Auto-research batches run in-process — durable on a long-lived server, but serverless deploys should move them to a QStash job. Health transitions over time ("who became dormant this week?") need a small snapshot table + cron, deliberately out of v1 scope.
